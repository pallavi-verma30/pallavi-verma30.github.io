A weekend bug that almost ate a pilot

A logic flaw was closing every case for a user the moment one was discontinued.

WherePilot deployment, Uganda
WhenSunday, mid-rollout
StakesActive field data collection
OutcomePilot stayed on track, no data lost

What happened

Field workers reported on a Sunday that cases were vanishing from their screens. The rest of the team was offline, and pausing for a day would have rippled into the rollout schedule.

My move

  • Traced the issue to a logic flaw: discontinuing one case was closing every case the user owned.
  • Documented the bug, asked the partner to pause data collection for the day, and looped the team in for Monday.
  • Patched the logic, reopened affected cases, and audited the same pattern across other apps in the portfolio. Caught the same flaw in two before it caused issues.
Data-quality judgment Owning the gap Resolved

Image resolution and the unverified records

A means of verification was failing by design before anyone noticed.

WhereChild-health campaign
When~1,000 records into rollout
StakesVerification of every entry
OutcomeProcess change, batch-validation now standard

What happened

Each child's vaccination was verified by an image of their immunization card. To keep field apps performant on low-end devices, image file size had been set low, but at that resolution the cards weren't readable. By the time I ran the data quality review, roughly a thousand records had already been collected without usable verification.

My move

  • Flagged the gap, paused collection, and ran a structured root-cause review with the partner.
  • Raised image resolution, tested across device tiers to confirm performance held, updated sync guidance for field users.
  • Owned the ~1,000-record gap in the post-mortem rather than papering over it. Now design against the problem by validating the data pipeline end-to-end against a small batch before scaling collection.

Independent build · 2026

AI-Powered Job
Application Tailoring System.

Job hunting in the age of AI is brutal. Hours to find roles that actually fit, then more hours editing resumes one-by-one. If résumé shortlisting on the employer side is already automated, why isn't the job-hunting side?

I'm an engineer at heart. Solving problems is my first instinct. And I'm not a developer by trade, so building this also meant learning a new skill: understanding what AI can and can't do, and how to build with it responsibly.

What it does

  1. Discovers job postings
  2. Scores match against base resume
  3. Tailors resume + cover letter for jobs scoring ≥75
  4. Renders 2-page A4 PDFs
  5. Sends a daily digest email of only high-fit matches

Stack

  • BackendPython · Flask · ~10 REST endpoints
  • Orchestrationn8n workflows in Docker
  • Data ingestionPlaywright (browser automation) · BeautifulSoup (HTML parse)
  • AIGemini 3.1 Flash Lite for scoring and tailoring
  • DocumentsWeasyPrint + Jinja2 (PDF) · pdfplumber (resume read) · openpyxl (Excel I/O)

Two ingestion modes

Automated ingestion from structured job aggregators, plus a manual URL batch mode where I curate URLs in an 8-column Excel and the same scoring/tailoring pipeline runs against each. Idempotent across re-runs via per-row status tracking, with a fetch-failed → manual-JD fallback path so the system stays useful on the days the ingestion layer breaks.

Quality controls

A 3-step structured prompt (key-term extraction → 0–100 scoring → tailored JSON output), anti-hallucination rules (banned strong-claim patterns, mandatory source-artifact naming in summaries, 4-item self-validation checklist), defensive placeholder scrubbing on Gemini output, and CSV-level deduplication via normalized URL matching.

Outputs

2-page A4 resume + cover letter PDFs per qualified job · daily digest email · complete audit log in jobs_log.csv (score, reasoning, source URL, PDF path).

Lessons learned

  1. Anti-hallucination is a design problem, not a prompt-tuning one. Models will quietly relabel candidate experience to match job-description language. Banned-pattern checks plus a self-validation pass caught what prompting alone could not.
  2. Narrow LLM responsibility produces better output. Splitting the job into discrete steps (extract key terms → score → tailor) returned more reliable JSON than asking one prompt for everything.
  3. AI-assisted development is real, but not magical. I leaned on Claude to write code I couldn't write from scratch, and still had to learn enough Docker, n8n, and Playwright to debug what it produced.

Hosting & privacy

Runs locally on-device. Not deployed publicly, by design, since it handles personal Gmail credentials and unfiltered job data. The architecture is portable and could be containerized for cloud deployment if a multi-user need arose.
